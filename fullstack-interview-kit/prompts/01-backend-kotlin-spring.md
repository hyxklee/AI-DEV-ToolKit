# Backend Prompt: Kotlin Spring Boot

Use this prompt when implementing backend code in a Kotlin Spring Boot live coding interview.

## Priority

Follow the target project's existing backend shape first. The kit provides quality defaults, not mandatory architecture.

## Architecture

Sprint/Core default:

```text
controller -> service -> repository -> entity
```

Full/Polish or existing rich architecture:

```text
presentation -> application -> domain <- infrastructure
```

## Detailed Rule Context

Read `.interview-rules.md` first. It is generated by `/setup` from the target project. If it is missing, infer conventions from nearby code instead of applying a bundled architecture.

## Rules

- Kotlin only.
- Controller handles HTTP only.
- Service or UseCase handles orchestration and transaction boundaries, depending on the target project.
- QueryService handles read-only DTO assembly only when the target project uses that pattern or the query is complex.
- Entity owns business rules and state changes.
- Repository is used directly by same-domain Service/UseCase.
- Cross-domain Reader interfaces and Port adapters are optional advanced structure; do not add them in Sprint/Core unless the project already has them.
- Do not create thin wrapper services such as GetService, SaveService, UpdateService, or DeleteService.
- Put `@Transactional` on Service, UseCase, or QueryService entry points, matching the project.
- DTOs are `data class`.
- JPA entities are regular `class` types, not `data class`.
- Prefer `val`, avoid `!!`, and use explicit exceptions for missing resources.
- Use `ResponseEntity` and plain exceptions unless the target project already has response wrappers and error-code types.

## Package Shape Options

Sprint/Core default:

```text
src/main/kotlin/{base}/{resource}/
  {Resource}Controller.kt
  {Resource}Service.kt
  {Resource}Repository.kt
  {Resource}.kt
  dto/
```

Use the target project's package shape if it exists.

Full/Polish reference shape:

```text
src/main/kotlin/{base}/domain/{domain}/
  application/dto/request/
  application/dto/response/
  application/mapper/
  application/usecase/command/
  application/usecase/query/
  application/exception/
  domain/entity/
  domain/repository/
  domain/service/
  domain/vo/
  domain/enums/
  domain/port/
  infrastructure/
  presentation/
```

## Checklist

1. Define request and response DTOs.
2. Define or update entity business methods.
3. Define repository methods with clear names.
4. Add Service or UseCase method with the transaction boundary.
5. Add mapper only if it removes real mapping noise or the project already uses mappers.
6. Add query service only for existing patterns or complex read assembly.
7. Add controller endpoint with validation and the project's response style.
8. Add focused tests for entity rules, service/usecase orchestration, or controller contract when time allows.
